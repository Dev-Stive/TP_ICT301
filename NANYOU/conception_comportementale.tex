\section{Section 1 : Vue d’ensemble du comportement}
Le système repose sur une architecture pilotée par les services. Le \textbf{Frontend} communique avec un \textbf{Backend} via une API REST. Chaque requête est interceptée par l'\textbf{AuthService} pour validation avant que les services métiers (\textbf{CahierService}, \textbf{AbsenceService}, \textbf{ReportService}) n'exécutent la logique et persistent les données via les \textbf{Repositories}. L'\textbf{AuditService} assure une traçabilité systématique de chaque action sensible.

\section{Section 2 : Scénarios critiques détaillés}

\subsection{Scénario 1 : Saisie d’une séance (Enseignant)}
\textbf{Description :} L'enseignant enregistre le contenu d'un cours.
\begin{enumerate}
    \item L'enseignant remplit le formulaire (Date, Horaire, Contenu).
    \item Le Frontend effectue une pré-validation client-side.
    \item \texttt{AuthService} vérifie l'identité et le rôle de l'enseignant.
    \item \texttt{CahierService} valide la cohérence métier des horaires.
    \item \texttt{SessionRepository} sauvegarde en base de données.
    \item \texttt{AuditService} enregistre l'action.
\end{enumerate}

\subsection{Scénario 2 : Consultation du cahier de texte (Étudiant)}
\textbf{Description :} L'étudiant consulte les détails d'une séance de sa classe.
\begin{enumerate}
    \item L'étudiant sélectionne une séance dans son emploi du temps.
    \item \texttt{AuthService} vérifie l'authentification.
    \item \texttt{CahierService} vérifie l'inscription de l'étudiant à la classe.
    \item \texttt{SessionRepository} récupère les données.
\end{enumerate}

\subsection{Scénario 3 : Validation d’une séance (Administrateur)}
\textbf{Description :} L'admin approuve ou rejette une séance saisie par un enseignant.
\begin{enumerate}
    \item L'admin examine la liste des séances en attente.
    \item \texttt{CahierService} met à jour le statut (validée ou rejetée).
    \item \texttt{NotificationService} alerte l'enseignant concerné par email.
\end{enumerate}

\subsection{Scénario 4 : Consultation des absences (Parent)}
\textbf{Description :} Le parent consulte l'historique des absences de son enfant.
\begin{enumerate}
    \item Le parent sélectionne l'enfant sur le Frontend.
    \item \texttt{AbsenceService} vérifie le lien de parenté.
    \item \texttt{AbsenceRepository} extrait la liste des absences filtrée.
\end{enumerate}

\subsection{Scénario 5 : Génération d’un rapport (Direction)}
\textbf{Description :} La direction génère des statistiques de couverture pédagogique.
\begin{enumerate}
    \item Le directeur sélectionne les filtres de période et de matière.
    \item \texttt{ReportService} agrège les données des sessions et absences.
    \item Le Backend génère un fichier PDF et le renvoie pour téléchargement.
\end{enumerate}

\section{Section 4 : Gestion des erreurs}
\begin{itemize}
    \item \textbf{401 (Unauthorized) :} Redirection vers la page de login.
    \item \textbf{403 (Forbidden) :} Message d'erreur "Permissions insuffisantes".
    \item \textbf{500 (Server Error) :} Échec système, proposition de "Réessayer" après 30 secondes.
\end{itemize}
